# M2 存储层与版本发布解读

本文用于精读 Smart-Workflow 的 M2 代码，覆盖：

- `migrations/00002_core_tables.sql`：核心表结构
- `internal/storage/mysql/store.go`：MySQL Store 封装
- `internal/storage/mysql/queries/*.sql`：sqlc 查询定义
- `internal/storage/mysql/gen/*.go`：sqlc 生成代码
- `internal/service/workflow.go`：WorkflowService 业务封装
- `internal/service/workflow_integration_test.go`：M2 集成测试

M2 的核心目标：

```text
把 M1 生成的 DSL 持久化到 MySQL；
支持草稿更新；
支持发布成不可变版本快照；
为 M4 执行引擎预留 workflow_run / node_run 运行记录表。
```

## 1. 先回答两个容易误解的问题

### 1.1 `workflow_version` 表使用了乐观锁吗？

没有。

当前代码里，乐观锁用在 `workflow` 表，不是 `workflow_version` 表。

对应 SQL 在 `internal/storage/mysql/queries/workflow.sql`：

```sql
-- name: UpdateWorkflowDraft :execresult
UPDATE workflow
SET name = ?, description = ?, draft_dsl = ?, version_lock = version_lock + 1
WHERE workflow_id = ? AND version_lock = ? AND deleted_at IS NULL;
```

这里的关键是：

```text
WHERE version_lock = ?
```

调用方必须带上自己读到的 `version_lock`。如果这期间别人已经更新过草稿，数据库里的 `version_lock` 就变了，这次 UPDATE 会影响 0 行，业务层返回 `ErrVersionLock`。

所以：

```text
workflow.version_lock 是草稿更新的乐观锁。
workflow_version.version 是发布快照的版本号，不是乐观锁。
```

### 1.2 为什么 `workflow_version` 不需要对“某条数据”加乐观锁？

因为 `workflow_version` 的设计语义是：

```text
只追加，不修改。
```

发布一次就插入一行：

```sql
INSERT INTO workflow_version (workflow_id, version, dsl, change_log)
VALUES (?, ?, ?, ?);
```

发布后的版本快照不应该再被更新。也就是说，正常业务里不存在：

```text
多个请求同时更新 workflow_version 的同一行
```

真正可能发生的并发是：

```text
两个请求同时 publish 同一个 workflow，尝试生成同一个新 version。
```

当前表结构用唯一键兜底：

```sql
UNIQUE KEY uk_wf_ver (workflow_id, version)
```

如果两个事务都算出 `newVer = 3`，只有一个能成功插入 `(workflow_id, 3)`，另一个会因为唯一键冲突失败。

因此更准确地说：

```text
workflow_version 不用乐观锁防“同一行更新”；
它用 append-only + 唯一键防“重复版本插入”。
```

### 1.3 当前 Publish 并发控制是否足够？

当前实现是 MVP 级别，可以工作，但不是最强并发控制。

当前 `Publish` 流程在 `internal/service/workflow.go`：

```go
wf, err := q.GetWorkflow(ctx, workflowID)
maxVerRaw, err := q.MaxWorkflowVersion(ctx, workflowID)
newVer = toInt32(maxVerRaw) + 1
q.CreateWorkflowVersion(... Version: newVer, Dsl: wf.DraftDsl ...)
q.PublishWorkflow(... PublishedVer: newVer ...)
```

它放在事务里，但 `GetWorkflow` 没有 `FOR UPDATE`，`MaxWorkflowVersion` 也没有显式锁。

并发 publish 时可能出现：

```text
事务 A 读到 max version = 2，准备插入 version 3
事务 B 也读到 max version = 2，准备插入 version 3
事务 A 插入成功
事务 B 插入失败：唯一键冲突
```

这不会产生重复版本，也不会污染已有版本，但其中一个发布请求会失败。

如果要做生产级增强，可以选：

```text
方案 1：发布前 SELECT workflow ... FOR UPDATE，锁住 workflow 主行
方案 2：用 workflow.published_ver + 1 生成版本，并在同一个行锁事务内更新
方案 3：捕获唯一键冲突后重试 MaxWorkflowVersion + Insert
```

当前代码还没有做这些增强。

### 1.4 通过 `version` 控制不同版本，是不是每次只能读取当前最新版本？

不是。

`version` 的意义是：

```text
每个发布版本都有一个明确编号，可以读取任意指定版本。
```

读取指定版本的 SQL 是：

```sql
-- name: GetWorkflowVersion :one
SELECT id, workflow_id, version, dsl, change_log, created_at
FROM workflow_version
WHERE workflow_id = ? AND version = ?;
```

业务层方法是：

```go
func (s *WorkflowService) GetVersion(ctx context.Context, workflowID string, version int32) (*dsl.DSL, error)
```

所以可以读：

```text
workflow_id=wf_abc, version=1
workflow_id=wf_abc, version=2
workflow_id=wf_abc, version=3
```

都可以。

`workflow.published_ver` 只是指向“当前最新发布版本”的指针。

```text
如果调用方想运行最新版本，就先读 workflow.published_ver。
如果调用方想运行历史版本，就直接指定 version。
如果调用方想调试草稿，M4 约定 version = -1 读取 draft_dsl。
```

也就是说：

```text
version 不是限制只能读最新；
version 是让你可以精确读取某一个版本。
```

## 3. 为什么 M2 存 DSL，不存 IR？

M1 里我们区分：

```text
IR  = Agent/CLI 编辑态
DSL = 引擎/存储执行态
```

M2 当前存的是 DSL：

```go
func (s *WorkflowService) Create(ctx context.Context, projectID, name, description string, draft *dsl.DSL) (string, error)
```

也就是 `draft *dsl.DSL`。

原因：

```text
1. 引擎只认 DSL，不认 IR。
2. PaiFlow 类设计也是 JSON DSL 存整图。
3. DSL 字段完整，发布快照能直接执行。
4. IR 更适合 CLI/session 文件和 Agent 小步编辑。
```

但注意一个命名细节：

```sql
draft_dsl JSON NOT NULL
```

这里的 `draft` 是“草稿状态”，不是 IR。

所以：

```text
workflow.draft_dsl = 可变草稿 DSL
workflow_version.dsl = 不可变发布 DSL
```

## 5. sqlc 是怎么工作的？

M2 不是在 Go 代码里手写 SQL 字符串，而是：

```text
queries/*.sql
  |
  | sqlc generate
  v
gen/*.go
```

例如：

```sql
-- name: GetWorkflow :one
SELECT id, workflow_id, project_id, name, description, draft_dsl,
       published_ver, status, version_lock, created_at, updated_at
FROM workflow
WHERE workflow_id = ? AND deleted_at IS NULL;
```

sqlc 会生成：

```go
func (q *Queries) GetWorkflow(ctx context.Context, workflowID string) (GetWorkflowRow, error)
```

好处：

```text
1. SQL 仍然由人明确控制。
2. Go 调用是强类型的。
3. JSON 列可以映射成 json.RawMessage。
4. 不引入 ORM 的隐式行为。
```

## 6. `mysql.Store` 封装

源码：`internal/storage/mysql/store.go`

核心结构：

```go
type Store struct {
    DB *sql.DB
    Q  *gen.Queries
}
```

含义：

| 字段 | 用途 |
|---|---|
| `DB` | 原生数据库连接池 |
| `Q` | sqlc 生成的查询集合 |

打开连接：

```go
func Open(dsn string) (*Store, error)
```

它会：

```text
1. sql.Open
2. 设置连接池参数
3. Ping 验证连接
4. 返回 Store{DB, Q}
```

事务封装：

```go
func (s *Store) WithTx(fn func(q *gen.Queries) error) error
```

调用方式：

```go
err := store.WithTx(func(q *gen.Queries) error {
    // 用事务内的 q 执行 SQL
    return nil
})
```

事务内的 `q` 来自：

```go
s.Q.WithTx(tx)
```

这样 service 层不用直接管理 `BEGIN/COMMIT/ROLLBACK`。

## 7. WorkflowService 是 M2 的业务门面

源码：`internal/service/workflow.go`

结构：

```go
type WorkflowService struct {
    store *mysql.Store
}
```

它封装工作流持久化操作：

```text
Create
Get
UpdateDraft
Publish
GetVersion
```

### 7.1 `Create`

代码：

```go
func (s *WorkflowService) Create(ctx context.Context, projectID, name, description string, draft *dsl.DSL) (string, error)
```

流程：

```text
1. 生成 workflow_id
2. 把 draft DSL marshal 成 JSON
3. INSERT workflow
4. 返回 workflow_id
```

关键代码：

```go
wfID := genID("wf")
raw, err := marshalDSL(draft)
q.CreateWorkflow(... DraftDsl: raw ...)
```

`marshalDSL`：

```go
func marshalDSL(d *dsl.DSL) (json.RawMessage, error) {
    if d == nil {
        return json.RawMessage("{}"), nil
    }
    b, err := json.Marshal(d)
    return b, err
}
```

也就是说：

```text
Go struct dsl.DSL -> JSON -> MySQL JSON 列
```

### 7.2 `Get`

代码：

```go
func (s *WorkflowService) Get(ctx context.Context, workflowID string) (*Workflow, error)
```

流程：

```text
1. SELECT workflow
2. 如果 sql.ErrNoRows，转成 ErrNotFound
3. draft_dsl 反序列化成 *dsl.DSL
4. 返回对外 Workflow 结构
```

关键代码：

```go
row, err := s.store.Q.GetWorkflow(ctx, workflowID)
draft, err := unmarshalDSL(row.DraftDsl)
```

`unmarshalDSL`：

```go
func unmarshalDSL(raw json.RawMessage) (*dsl.DSL, error) {
    if len(raw) == 0 || string(raw) == "{}" || string(raw) == "null" {
        return &dsl.DSL{}, nil
    }
    var d dsl.DSL
    json.Unmarshal(raw, &d)
    return &d, nil
}
```

也就是说：

```text
MySQL JSON 列 -> json.RawMessage -> Go struct dsl.DSL
```

### 7.3 `UpdateDraft`

代码：

```go
func (s *WorkflowService) UpdateDraft(ctx context.Context, workflowID, name, description string, draft *dsl.DSL, expectLock int32) error
```

流程：

```text
1. marshal 新草稿 DSL
2. UPDATE workflow
3. WHERE workflow_id = ? AND version_lock = ?
4. 如果影响 0 行，返回 ErrVersionLock
```

对应 SQL：

```sql
UPDATE workflow
SET name = ?, description = ?, draft_dsl = ?, version_lock = version_lock + 1
WHERE workflow_id = ? AND version_lock = ? AND deleted_at IS NULL;
```

这就是乐观锁。

#### 乐观锁例子

假设当前：

```text
workflow.version_lock = 0
```

Agent A 和 Agent B 同时读到：

```text
version_lock = 0
```

Agent A 更新：

```sql
WHERE version_lock = 0
```

成功，数据库变成：

```text
version_lock = 1
```

Agent B 继续拿旧 lock 更新：

```sql
WHERE version_lock = 0
```

影响 0 行，服务返回：

```go
ErrVersionLock
```

这避免 B 覆盖 A 的更新。

### 7.4 `Publish`

代码：

```go
func (s *WorkflowService) Publish(ctx context.Context, workflowID, changeLog string) (int32, error)
```

流程：

```text
1. 开启事务
2. 读取 workflow 当前草稿
3. 查询 workflow_version 当前最大 version
4. newVer = max + 1
5. 插入 workflow_version，dsl = 当前 draft_dsl
6. 更新 workflow.published_ver = newVer，status = 已发布
7. 提交事务
```

关键代码：

```go
wf, err := q.GetWorkflow(ctx, workflowID)
maxVerRaw, err := q.MaxWorkflowVersion(ctx, workflowID)
newVer = toInt32(maxVerRaw) + 1

q.CreateWorkflowVersion(ctx, gen.CreateWorkflowVersionParams{
    WorkflowID: workflowID,
    Version:    newVer,
    Dsl:        wf.DraftDsl,
    ChangeLog:  changeLog,
})

q.PublishWorkflow(ctx, gen.PublishWorkflowParams{
    PublishedVer: newVer,
    WorkflowID:   workflowID,
})
```

关键语义：

```text
Publish 不是把 draft_dsl 改名为 version。
Publish 是复制当前 draft_dsl，插入 workflow_version 形成快照。
```

发布后继续改草稿，不会影响已经发布的版本。

### 7.5 `GetVersion`

代码：

```go
func (s *WorkflowService) GetVersion(ctx context.Context, workflowID string, version int32) (*dsl.DSL, error)
```

流程：

```text
1. SELECT workflow_version WHERE workflow_id=? AND version=?
2. 找不到返回 ErrNotFound
3. dsl JSON 反序列化成 *dsl.DSL
```

它读取的是指定版本，不一定是最新版本。

## 8. 草稿、发布版本、运行版本的关系

可以用这张图理解：

```text
workflow.draft_dsl
  当前草稿，可变
  |
  | Publish
  v
workflow_version.dsl
  发布快照，不可变
  version = 1 / 2 / 3 ...
  |
  | engine.Run
  v
workflow_run.version
  记录这次运行用的是哪个版本
```

更具体：

```text
用户编辑草稿：
  UPDATE workflow.draft_dsl

用户发布：
  INSERT workflow_version(version=N, dsl=当前draft_dsl)
  UPDATE workflow.published_ver=N

用户运行：
  创建 workflow_run(workflow_id, version=N)
  engine.Run 根据 run.version 读取 workflow_version.dsl
```

如果是草稿调试：

```text
workflow_run.version = -1
engine.Run 读取 workflow.draft_dsl
```

这个约定在 M4 中使用。

## 9. 是否只能运行最新版本？

不是。

有三种常见模式：

### 9.1 运行最新发布版本

流程：

```text
1. GetWorkflow 读取 published_ver
2. 创建 workflow_run(version=published_ver)
3. engine.Run 读取 workflow_version.dsl
```

### 9.2 运行指定历史版本

流程：

```text
1. 用户指定 version=1
2. 创建 workflow_run(version=1)
3. engine.Run 读取 workflow_version where version=1
```

### 9.3 运行当前草稿

流程：

```text
1. 创建 workflow_run(version=-1)
2. engine.Run 读取 workflow.draft_dsl
```

所以：

```text
published_ver 只是默认“最新版本指针”。
version 表示可精确定位某个发布快照。
```

## 10. 软删除设计

M2 使用软删除：

```sql
UPDATE workflow
SET deleted_at = NOW()
WHERE workflow_id = ? AND deleted_at IS NULL;
```

查询时过滤：

```sql
WHERE workflow_id = ? AND deleted_at IS NULL;
```

好处：

```text
1. 删除后业务不可见。
2. 数据仍保留，方便审计和恢复。
3. 不破坏历史 run/version 关系。
```

注意：

```text
workflow_version 当前没有 deleted_at。
因为版本快照是历史记录，通常不随 workflow 软删除而物理删除。
```

## 12. 当前 M2 的设计边界和后续增强

当前已经完成：

```text
1. MySQL 表结构
```

当前还可以增强：

```text
1. Publish 并发时加 workflow 行锁，避免唯一键冲突暴露给用户。
2. Publish 支持 expectLock，确保发布的是用户刚确认的草稿版本。
3. workflow_version 增加发布人、发布来源、hash 等审计字段。
4. workflow_run 增加 trace_id / request_id，方便排查。
5. node_run 对 output 大对象做裁剪或对象存储引用。
```

其中最值得优先增强的是发布并发控制：

```sql
SELECT ... FROM workflow WHERE workflow_id = ? FOR UPDATE;
```

或者在 service 层对唯一键冲突做重试。

## 13. 一句话总结 M2

```text
M2 把 DSL 存入 MySQL：workflow 存可变草稿，workflow_version 存不可变发布快照，version_lock 只保护草稿并发更新，version 用来精确读取任意发布版本，而不是只能读取最新版本。
```
